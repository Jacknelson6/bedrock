# SSR & Prerender Patterns — Bedrock Reference

Every Bedrock component must be implemented with SEO and AEO (Answer Engine Optimization) in mind. Animated UIs are useless if search engines can't index the content. This document defines the patterns that make animated pages prerender-friendly.

## The Core Problem

Motion animations require client-side JavaScript. Search engine crawlers (Googlebot, Bingbot) and AI answer engines (Perplexity, ChatGPT Browse, Google AI Overviews) may or may not execute JavaScript. The solution: **render meaningful content server-side, enhance with animation client-side.**

## Universal Principles

### 1. Content-First Rendering
Every animated component should render its FINAL STATE in the initial HTML. The animation is a progressive enhancement, not a prerequisite for content visibility.

```tsx
// ❌ BAD — content invisible until JS runs
<motion.div initial={{ opacity: 0 }} animate={{ opacity: 1 }}>
  <h1>Our Product</h1>
</motion.div>

// ✅ GOOD — content visible in HTML, animation enhances
<motion.div
  initial={{ opacity: 0, y: 20 }}
  animate={{ opacity: 1, y: 0 }}
  style={{ opacity: 1 }} // SSR fallback: visible by default
>
  <h1>Our Product</h1>
</motion.div>
```

### 2. The `initial={false}` Pattern
For content that must be visible on first paint (hero headlines, above-the-fold content), use `initial={false}` to skip the entrance animation on SSR while still enabling it on client navigation:

```tsx
'use client';
import { useReducedMotion } from 'motion/react';

function HeroHeadline({ text }: { text: string }) {
  const prefersReduced = useReducedMotion();

  return (
    <motion.h1
      initial={prefersReduced ? false : { opacity: 0, y: 30 }}
      animate={{ opacity: 1, y: 0 }}
      transition={{ duration: 0.6, ease: "easeOut" }}
    >
      {text}
    </motion.h1>
  );
}
```

### 3. Semantic HTML Always
Animated wrappers must not replace semantic elements. The crawler should see proper `<h1>`, `<h2>`, `<p>`, `<section>`, `<article>`, `<nav>` structure regardless of animation.

```tsx
// ❌ BAD — loses semantics
<SplitText text="Our Product Features" />

// ✅ GOOD — semantics preserved, wrap in semantic element
<h2 className="text-3xl font-bold">
  <SplitText text="Our Product Features" />
</h2>
```

### 4. Text Content Must Be in the DOM
Never hide text content behind JavaScript-only rendering. All text that should be indexed must exist in the server-rendered HTML.

```tsx
// ❌ BAD — text generated by JS animation (DecryptedText producing characters)
// The crawler sees random characters, not your content

// ✅ GOOD — real text in DOM, animation is visual enhancement
<span aria-label="Welcome to our platform">
  <DecryptedText text="Welcome to our platform" />
</span>
```

## Framework-Specific Patterns

### Next.js App Router (Primary Target)

**Component Architecture:**
```
app/
├── page.tsx              ← Server Component (SEO content, metadata)
├── layout.tsx            ← Server Component (structure)
└── components/
    ├── hero.tsx          ← 'use client' (animated)
    ├── features.tsx      ← 'use client' (animated)
    └── content.tsx       ← Server Component (static content)
```

**The Split Pattern:** Keep the page itself as a Server Component. Import client components for animated sections:

```tsx
// app/page.tsx — SERVER COMPONENT (crawlable)
import { Metadata } from 'next';
import Hero from '@/components/hero';
import Features from '@/components/features';
import Pricing from '@/components/pricing';

export const metadata: Metadata = {
  title: 'Your SaaS Product — Tagline',
  description: 'Product description for SEO...',
  openGraph: { /* ... */ },
};

export default function HomePage() {
  return (
    <main>
      {/* Content exists in HTML for crawlers */}
      <Hero
        headline="Build Faster, Ship Sooner"
        subheadline="The development platform that..."
        ctaText="Get Started Free"
        ctaHref="/signup"
      />
      <Features features={featuresData} />
      <Pricing plans={pricingData} />
    </main>
  );
}
```

```tsx
// components/hero.tsx — CLIENT COMPONENT (animated)
'use client';
import { motion } from 'motion/react';
import { SplitText } from '@/components/ui/split-text';
import { Magnet } from '@/components/ui/magnet';

interface HeroProps {
  headline: string;
  subheadline: string;
  ctaText: string;
  ctaHref: string;
}

export default function Hero({ headline, subheadline, ctaText, ctaHref }: HeroProps) {
  return (
    <section className="relative min-h-screen flex items-center justify-center">
      {/* Text content is always in the DOM for SEO */}
      <div className="text-center max-w-4xl mx-auto px-4">
        <h1 className="text-6xl font-bold">
          <SplitText text={headline} delay={50} />
        </h1>
        <motion.p
          initial={{ opacity: 0, y: 20 }}
          animate={{ opacity: 1, y: 0 }}
          transition={{ delay: 0.3, duration: 0.5 }}
          className="mt-6 text-xl text-muted-foreground"
        >
          {subheadline}
        </motion.p>
        <Magnet>
          <motion.a
            href={ctaHref}
            initial={{ opacity: 0, y: 20 }}
            animate={{ opacity: 1, y: 0 }}
            transition={{ delay: 0.6, duration: 0.5 }}
            className="inline-block mt-8 px-8 py-4 bg-primary text-primary-foreground rounded-full"
          >
            {ctaText}
          </motion.a>
        </Magnet>
      </div>
    </section>
  );
}
```

**Key point:** The `headline`, `subheadline`, and `ctaText` props are passed from the Server Component. Even though the Hero is a client component, Next.js serializes these props into the HTML during SSR. The crawler sees the text content.

### Metadata & Structured Data

Every page should include comprehensive metadata:

```tsx
export const metadata: Metadata = {
  title: 'Page Title — Brand',
  description: 'Clear, keyword-rich description (150-160 chars)',
  keywords: ['relevant', 'keywords'],
  openGraph: {
    title: 'Page Title — Brand',
    description: 'Social sharing description',
    images: [{ url: '/og-image.png', width: 1200, height: 630 }],
    type: 'website',
  },
  twitter: {
    card: 'summary_large_image',
    title: 'Page Title — Brand',
    description: 'Social sharing description',
    images: ['/og-image.png'],
  },
  alternates: {
    canonical: 'https://yourdomain.com/page',
  },
};
```

### JSON-LD Structured Data

For AEO (Answer Engine Optimization), add JSON-LD:

```tsx
// app/page.tsx
export default function HomePage() {
  const jsonLd = {
    '@context': 'https://schema.org',
    '@type': 'SoftwareApplication',
    name: 'Your Product',
    description: 'What your product does',
    applicationCategory: 'BusinessApplication',
    operatingSystem: 'Web',
    offers: {
      '@type': 'Offer',
      price: '0',
      priceCurrency: 'USD',
    },
  };

  return (
    <>
      <script
        type="application/ld+json"
        dangerouslySetInnerHTML={{ __html: JSON.stringify(jsonLd) }}
      />
      <main>{/* ... */}</main>
    </>
  );
}
```

## Component-Level SSR Patterns

### Text Effects — All Prerender-Safe
All text components render their text content into the DOM. The animation is visual enhancement only. Crawlers see the final text.

**Exception:** DecryptedText and ScrambledText show random characters during animation. Wrap with `aria-label` to ensure the real text is accessible:
```tsx
<span aria-label="Your actual text here">
  <DecryptedText text={text} />
</span>
```

### Backgrounds — No SSR Concerns
Background components are decorative and don't contain indexable content. They render as empty containers server-side, which is correct behavior.

**Three.js backgrounds MUST use dynamic import:**
```tsx
const Particles = dynamic(() => import('@/components/particles'), {
  ssr: false,
  loading: () => <div className="absolute inset-0 bg-background" />,
});
```

### Interactive Elements — Graceful Degradation
Interactive elements (Magnet, GlareHover, ElectricBorder) do nothing server-side but render their children normally. This is correct — the content is visible, and interactivity is a progressive enhancement.

### Scroll-Triggered Entrances — Critical Pattern
Components like AnimatedContent and FadeContent hide their children initially. For SEO, ensure content is in the DOM:

```tsx
// The content is in the DOM but visually hidden until scrolled into view
// Crawlers still see and index the content — CSS opacity doesn't affect indexing
<FadeContent blur direction="up">
  <section>
    <h2>Your Indexable Content</h2>
    <p>This text is in the HTML source and will be indexed.</p>
  </section>
</FadeContent>
```

**Important:** Google does NOT penalize content that starts at `opacity: 0` as long as it's in the DOM and becomes visible on scroll. This is standard practice on virtually every modern marketing site.

### 3D/WebGL Components — Always SSR: false
```tsx
import dynamic from 'next/dynamic';

const Globe = dynamic(() => import('@/components/globe'), {
  ssr: false,
  loading: () => (
    // Provide meaningful fallback that crawlers see
    <div
      className="w-full h-[500px] flex items-center justify-center bg-muted rounded-lg"
      aria-label="Interactive 3D globe showing global presence"
    >
      <p className="text-muted-foreground">Loading interactive globe...</p>
    </div>
  ),
});
```

## Performance & Core Web Vitals

Animated pages must still pass Core Web Vitals for SEO ranking:

### LCP (Largest Contentful Paint)
- Hero text should animate FAST (< 300ms delay)
- Use `priority` on hero images: `<Image priority />`
- Don't block LCP with heavy animation libraries — Motion is only loaded client-side

### CLS (Cumulative Layout Shift)
- All animated containers must have explicit dimensions
- Use `min-h-screen` or fixed heights on hero sections
- `tabular-nums` on animated numbers to prevent width shifts
- Skeleton components must match final content dimensions

### INP (Interaction to Next Paint)
- Motion animations run off the main thread where possible (WAAPI)
- Don't animate on every scroll event — use Intersection Observer (FadeContent or useInView)
- Debounce cursor-following effects on mobile

### Bundle Splitting
```tsx
// Split heavy components into separate chunks
const HeavyBackground = dynamic(() => import('@/components/particles'), {
  ssr: false,
});

// Preload critical animation library
<link rel="modulepreload" href="/_next/static/chunks/motion.js" />
```

## Prerender / Static Generation Compatibility

All Bedrock components are compatible with:
- **Next.js SSG** (`generateStaticParams`)
- **Next.js ISR** (revalidation)
- **Static export** (`output: 'export'`)
- **Prerender** services (Prerender.io, Rendertron)

The only exceptions are Three.js components (Hyperspeed, Threads, Particles, Silk, Galaxy) which must be excluded from SSR. With `dynamic({ ssr: false })`, they simply don't render during static generation and hydrate client-side.

## AEO (Answer Engine Optimization) Checklist

For pages that should surface in AI-generated answers:

1. ✅ Clear, factual `<title>` and `<meta description>`
2. ✅ Proper heading hierarchy (`h1` → `h2` → `h3`)
3. ✅ JSON-LD structured data (SoftwareApplication, FAQPage, etc.)
4. ✅ Content visible in server-rendered HTML (not hidden behind JS)
5. ✅ FAQ section with `<details>/<summary>` or Accordion (both provide content in HTML)
6. ✅ Canonical URL set
7. ✅ Open Graph and Twitter Card metadata
8. ✅ Fast page load (< 3s) — animations don't block content paint
9. ✅ Mobile-responsive (all components use responsive Tailwind classes)
10. ✅ Descriptive `alt` text on all images, `aria-label` on decorative animations
